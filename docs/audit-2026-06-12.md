## Hallazgos

### 🔴 BLOQUEANTE — Brecha de seguridad RLS en `connections`

**Archivo:** `src/db/migrations/0002_row_level_security.sql` líneas 497–517  
**Tipo:** Security — bypass de regla de negocio a nivel de base de datos

#### Problema

La política `connections_update_participant_or_admin` permite a cualquier participante (tanto el requester como el recipient) actualizar cualquier campo del registro, incluyendo el campo `status`. Esto significa que un requester puede auto-aceptar su propia solicitud de conexión llamando directamente al cliente Supabase desde el navegador, sin pasar por la API route de la aplicación.

La API route en `src/app/api/network/connections/[connectionId]/route.ts` sí aplica la regla de negocio correcta (solo el recipient puede aceptar/rechazar, solo el requester puede cancelar), pero esta protección existe **únicamente a nivel de aplicación**. Cualquier llamada directa al SDK de Supabase la elude.

#### Política actual (vulnerable)

```sql
create policy "connections_update_participant_or_admin"
on public.connections for update
to authenticated
using (
  requester_profile_id = auth.uid()
  or recipient_profile_id = auth.uid()
  or public.current_profile_is_admin()
)
with check (
  requester_profile_id = auth.uid()
  or recipient_profile_id = auth.uid()
  or public.current_profile_is_admin()
);
```

#### Corrección: nueva migración `0005_fix_connections_rls.sql`

Eliminar la política genérica y reemplazarla por tres políticas con semántica explícita:

```sql
-- Eliminar política genérica vulnerable
drop policy if exists "connections_update_participant_or_admin" on public.connections;

-- Solo el recipient puede aceptar o rechazar una solicitud pendiente
create policy "connections_update_recipient_accept_reject"
on public.connections for update
to authenticated
using (
  recipient_profile_id = auth.uid()
  and status = 'pending'
)
with check (
  recipient_profile_id = auth.uid()
  and status in ('accepted', 'rejected')
);

-- Solo el requester puede cancelar una solicitud pendiente
create policy "connections_update_requester_cancel"
on public.connections for update
to authenticated
using (
  requester_profile_id = auth.uid()
  and status = 'pending'
)
with check (
  requester_profile_id = auth.uid()
  and status = 'cancelled'
);

-- El admin puede actualizar cualquier conexión
create policy "connections_update_admin"
on public.connections for update
to authenticated
using (public.current_profile_is_admin())
with check (public.current_profile_is_admin());
```

### 🟠 ALTO — Feed: carga de comentarios sin límite por post

**Archivo:** `src/lib/feed/queries.ts` líneas 66–72  
**Tipo:** Rendimiento — riesgo de OOM y timeout a escala

#### Problema

La query de comentarios en `getFeedPage` usa `.in("post_id", postIds)` sin `.limit()`. Para una página del feed de 10 posts, se cargan **todos** los comentarios de todos los posts. Si un post tiene 500 comentarios, se cargan los 500 en cada renderizado del feed de cualquier usuario.

```ts
// Actual — sin límite
supabase
  .from("comments")
  .select("id, post_id, author_profile_id, body, status, created_at, updated_at")
  .eq("status", "published")
  .in("post_id", postIds)
  .order("created_at", { ascending: true })
```

#### Corrección

Añadir `.limit()` en la query del feed y hacer que la carga completa de comentarios sea una llamada separada por post:

```ts
// En getFeedPage — máximo 3 comentarios preview por post
supabase
  .from("comments")
  .select("id, post_id, author_profile_id, body, status, created_at, updated_at")
  .eq("status", "published")
  .in("post_id", postIds)
  .order("created_at", { ascending: false })
  .limit(3 * postIds.length) // aproximación; idealmente window function por post
```

La UI debe mostrar los primeros 3 comentarios con un CTA "Ver todos los comentarios (N)" que dispare una llamada a `GET /api/feed/posts/[postId]/comments`.

### 🟠 ALTO — `getAppShellAuth`: queries secuenciales en cada render

**Archivo:** `src/lib/auth/session.ts` líneas 86–156  
**Tipo:** Rendimiento — latencia acumulada en cada carga de página autenticada

#### Problema

En cada página autenticada se ejecutan en cascada estas 4 queries Supabase:

1. `supabase.auth.getUser()`
2. `.from("profiles").select(...)`
3. `.from("companies")` o `.from("training_providers")` (dependiendo del tipo de perfil)
4. `.from("contact_requests").count`

Las queries 3 y 4 son completamente independientes entre sí y podrían ejecutarse en paralelo. Con 50ms por round-trip a Supabase, la diferencia es ~50ms adicionales en cada carga de página autenticada (query 3 o 4 en serie en lugar de paralelo).

#### Corrección

```ts
// Después de resolver el perfil, paralelizar la query de entidad y el count
const [entityResult, contactResult] = await Promise.all([
  profile?.profile_type === "company"
    ? supabase.from("companies").select("id, logo_url").eq("owner_profile_id", profile.id).maybeSingle()
    : profile?.profile_type === "training_provider"
      ? supabase.from("training_providers").select("id, logo_url").eq("owner_profile_id", profile.id).maybeSingle()
      : Promise.resolve({ data: null }),
  profile?.id
    ? supabase.from("contact_requests").select("id", { count: "exact", head: true })
        .eq("recipient_profile_id", profile.id).is("read_at", null).is("archived_at", null)
    : Promise.resolve({ count: 0 }),
]);
```

### 🟡 MEDIO — Paginación offset en el feed (deuda técnica conocida)

**Archivo:** `src/lib/feed/queries.ts` líneas 39–47  
**Tipo:** Escalabilidad — degradación progresiva con el volumen de posts

#### Problema

`.range(from, to)` obliga a Postgres a escanear y descartar `(page - 1) × FEED_PAGE_SIZE` filas en cada request. En página 50 con alto volumen de posts, el coste es equivalente a un full scan efectivo.

Este punto está anotado explícitamente en `CLAUDE.md` como "aceptable para validación MVP".

#### Acción requerida antes de pruebas de carga

Migrar a cursor-based pagination usando `created_at`:

```ts
// En lugar de .range(from, to):
.lt("created_at", lastSeenCreatedAt)  // cursor del último post visto
.limit(FEED_PAGE_SIZE)
```

La respuesta del API debe devolver el `created_at` del último item para que el cliente lo use como cursor en la siguiente página.

### 🟡 MEDIO — `as never[]` como escape hatch de tipos en 25 routes

**Archivos:** `src/app/api/**/route.ts` (~25 ficheros)  
**Tipo:** Deuda técnica de tipos

#### Problema

El patrón `supabase.from("x").insert([payload] as never[])` (y variantes `as never`) está presente en todos los routes de inserción y actualización. Es un workaround para una incompatibilidad de tipos entre `@supabase/supabase-js` y el App Router de Next.js. No causa errores en runtime pero silencia los errores de tipo en los payloads de operaciones de escritura.

#### Acción

Abrir un ticket de deuda técnica. Cuando `@supabase/supabase-js` publique una versión compatible con los tipos generados de Next.js, eliminar todos los `as never[]` en un único commit de limpieza.

### 🟢 BAJO — Sin infraestructura de tests

**Tipo:** Calidad — riesgo creciente con la superficie de API

No existe ningún fichero de test ni configuración de Jest, Vitest o Playwright en el repositorio. Para MVP es una decisión válida. Sin embargo, con más de 30 API routes ya implementados, la ausencia de tests hace que los cambios de lógica de negocio (especialmente en auth, RLS y el feed) sean difíciles de validar de forma automatizada.

**Recomendación mínima:** Tests de integración para los flujos de auth (sign-in, OAuth callback, onboarding) y para las operaciones críticas del feed (crear post, like, comentario) usando Vitest con un cliente Supabase contra un proyecto de staging.
